## 目标（给我自己/未来改代码用）

把 `interact_with_terminal` 的“等待/轮询”能力做得更省调用次数、更省 token，并且避免 AI 因为误判“命令会结束”而卡死等待。

重点围绕：
- **减少轮询次数**：长命令（build/test）不需要反复 `waitForOutput` 拉 `tail/raw` 才知道是否结束。
- **减少输出 token**：避免重复回显、提示符刷屏、同一段输出在多种 readMode 里重复返回。
- **安全**：永远不要让 AI 默认使用“等进程退出”这种可能无限等待的策略。

---

## 现状定位（源码位置）

### 1) 现有“等待输出稳定”
- 文件：`src/terminal-manager.ts`
- 方法：`waitForOutputStable(terminalName, timeout=5000, stableTime=500)`（大约在 `~991` 行附近）
- 行为：轮询 `session.lastActivity`，如果 `stableTime` 内无新输出就返回；超时也返回但不报错。

### 2) 现有“读取输出/游标”
- 文件：`src/terminal-manager.ts`
- 方法：`readFromTerminal(options)`（大约在 `~498` 行附近）
- 已有字段：`since`/`cursor`、`totalLines`、`hasMore`、`truncated`、`stats.totalBytes/estimatedTokens`、`status` 等。
- 已有模式：`smart/auto/full/raw/head/tail/head-tail`（具体看 `selectReadMode` 与相关分支）。
- 已做的优化：针对 alternate screen / ANSI 重绘，倾向从 `rawOutput` 渲染快照，减少“行缓冲不准确”问题。

### 3) MCP 工具层参数与行为
- 文件：`src/mcp-server.ts`
- 工具：`interact_with_terminal`
- 入参：`waitForOutput` 当前是“秒数”（`wait time in seconds`），并且有 `mode/maxLines/since/tailLines/stripSpinner/search...` 等
- 行为：写入 input 后，会调用 `readFromTerminal`；另外会把 `structuredContent.status.pendingCommand/lastCommand/hasPrompt` 等返回给客户端。

## 问题总结（从本次使用体验 + 现有实现推导）

1) **等待策略单一**：`waitForOutputStable` 只按“输出静默”判断；而 MCP 入参叫 `waitForOutput`（秒），语义不够明确（是等输出出现？等静默？等命令结束？）。
2) **AI 无法可靠判断命令是否会结束**：如果提供“wait until exit”，AI 误用会在常驻进程（server/watch/tail -f）上卡住。
3) **输出重复/混杂导致 token 浪费**：
   - 终端提示符、命令回显、重复片段会占掉大量 token。
   - `smart/tail/raw` 切换时容易取到重叠内容，客户端只能靠多次调用和肉眼判断。
4) **缺少增量语义**：虽然底层已有 `since/cursor`，但工具层没有强调“默认增量返回”，也缺少“只返回新增输出”的强保证与便利字段（例如 `deltaText`）。

---

## 设计原则（必须遵守）

1) **默认不允许无限等待**：任何等待都必须有 `timeoutMs`，超时就返回“仍在运行 + 本次增量输出 + 状态”，而不是挂起。
2) **默认不使用 waitFor=exit**：只有在“用户显式指定”或“明确可判断场景”才允许。
3) **优先增量返回**：减少重复输出的 token 消耗；需要全文/快照时再显式请求。
4) **服务端给出可机器读取的状态字段**：让 AI 不用猜（是否在跑/是否在等待输入/是否进入 alternate screen/最后活动时间/退出码等）。

## 需要新增/调整的 MCP 入参（建议）

在 `interact_with_terminal` 新增一个等待策略对象（兼容旧字段）：

### 方案 A（推荐）：新增 `wait` 对象
新增：
- `wait?: {`
  - `mode: 'none' | 'idle' | 'prompt' | 'pattern' | 'exit'`
  - `timeoutMs: number`（必填，建议默认 1500~3000）
  - `idleMs?: number`（mode=idle 时用，默认 800~1500）
  - `pattern?: string`（mode=pattern 时用）
  - `patternRegex?: boolean`（默认 false）
  - `patternCaseSensitive?: boolean`（默认 false）
  - `includeIntermediateOutput?: boolean`（默认 true：等待期间持续累积增量输出，最终一起返回）
`}`

兼容：
- 旧的 `waitForOutput`（秒）仍保留：如果用户传了它但没传 `wait`，则映射为 `wait.mode='idle'`、`wait.timeoutMs = waitForOutput*1000`、`wait.idleMs=500`。

### 方案 B：直接扩展 `waitForOutput`
把 `waitForOutput` 从 number 改成 union（不推荐：破坏兼容较大）：
- `waitForOutput?: number | { ...同上... }`

## 服务端输出（structuredContent）需要增强的字段

在 `interact_with_terminal` 返回的 `structuredContent` 里补齐/强化这些字段，减少客户端再拉取/再猜：

- `commandId`: string（本次“写入 input”的唯一 ID；无 input 时可为空）
- `write`: `{ appendedNewline: boolean; bytesWritten: number; startedAt: string }`
- `read`: `{ mode: string; since: number; cursor: number; hasMore: boolean; truncated: boolean }`
- `delta`: `{ text: string; bytes: number; lines: number }`
  - 其中 `delta.text` 要尽量只包含 `[since, cursor]` 的新增内容，避免重复。
- `wait`: `{ mode: string; timeoutMs: number; met: boolean; reason: 'timeout'|'idle'|'prompt'|'pattern'|'exit'|'none' }`
- `status`: 保留现有字段，并建议新增：
  - `awaitingInput: boolean`（可直接复用 `terminalManager.isTerminalAwaitingInput()`）
  - `isLikelyInteractive: boolean`（复用 `isLikelyInteractiveTail` 或类似逻辑）
  - `lastOutputAt: string`（从 `session.lastActivity` 生成）
  - `exitCode?: number`（如果已退出）

## 终端读取/增量返回的落地改法（源码级提示）

### 1) 在 `TerminalManager.readFromTerminal` 强化“增量”语义
文件：`src/terminal-manager.ts`

建议新增一个模式（或参数）：
- `deltaOnly?: boolean`（默认 true），含义：尽量只返回自 `since` 以来新增内容，不要“为了凑 smart 模式”把旧内容也带上。

实现要点：
- 对于 `raw`/快照渲染分支：目前会取 `rawOutput` 的 tail（例如 8000/120000 chars），这会天然包含很多旧内容，导致 delta 不纯。
  - 改法：在 session 里维护 `rawSequenceCursor -> rawOutputIndex` 的映射（或保存每次 append 的 raw 长度与 sequence），从而能按 `since` 截取更接近增量的 raw 段。
  - 如果映射成本太高：至少在 `structuredContent` 里明确标记 `read.isDeltaAccurate=false`，让客户端知道这次是“快照/近似”而不是增量。

### 2) 把“等待条件”放到服务端执行，而不是客户端轮询
文件：`src/mcp-server.ts`（`interact_with_terminal`）

改造点：
- 写 input 前：记录 `cursorBeforeWrite`（当前 outputBuffer 最新 sequence）
- 写入后：进入一个循环：
  - 每 100~200ms 读取一次 `readFromTerminal({ since: cursorBeforeWrite, mode, ... })` 的 **增量**（累计到 `delta` buffer）
  - 同时检查 `wait.mode`：
    - `idle`：`Date.now()-lastActivity > idleMs`
    - `pattern`：在累计的 `delta.text`（或新读到的片段）里做匹配
    - `prompt`：复用 `status.hasPrompt`（或更严格：检测 prompt 行/terminal awaiting input）
    - `exit`：仅当用户显式要求时才检查 `session.status==='terminated'` 或等待 exit 事件
  - 命中则返回；否则超时返回（`wait.reason='timeout'`）

注意：**对常驻进程**（server/watch）通常会在启动后进入 idle，因此 `idle` 模式对 AI 默认最安全。

## 减少 token 的“去噪/折叠”策略（服务端）

目标：让默认返回更“干净”，必要时再让客户端显式要 raw。

建议在 `readFromTerminal` 或 `interact_with_terminal` 末尾新增一个轻量后处理（可控开关）：
- `normalizeOutput?: boolean`（默认 true）
  - 折叠连续提示符行（`PS ...>` 重复）
  - 折叠重复回显命令（同一 `input` 被回显多次时只保留第一次）
  - 折叠空行（连续超过 N 行）
  - 对 spinner/进度条：已有 `stripSpinner`，继续沿用并确保默认开启

并且把“去噪前/后”的统计差异写入 `structuredContent.stats`，方便评估 token 节省效果。

## “不要让 AI 猜”的机制（建议服务端直接返回决策信号）

在 `structuredContent.status` 里加入：
- `recommendedWaitMode: 'idle'|'prompt'|'none'`
- `recommendationReason: string`

逻辑示例：
- 如果 `awaitingInput=true` 或 `hasPrompt=true`：推荐 `prompt`
- 如果 `isLikelyInteractive=true`：推荐 `idle`（短 idleMs）
- 如果 `pendingCommand` 存在：推荐 `idle`

这样 AI 即使不知道命令是否会结束，也能“按建议字段”选策略而不是瞎猜。

---

## 验收标准（改完必须满足）

1) 对 build/test（会结束）：
   - 单次调用设置 `wait.mode='idle'`，能在命令结束后 1~2 秒内返回结果；
   - 不需要客户端再额外 `tail/raw` 拉取才能看到 `ok/FAIL`。
2) 对常驻进程（不会结束，例如启动 server）：
   - 单次调用不会卡死：超时返回，并且 `status.isRunning=true`；
   - 后续再次调用时，只返回新增输出（delta），不会重复刷历史。
3) token 明显下降：
   - 相同场景下，默认返回不包含大量重复提示符/命令回显；
   - `structuredContent.stats` 能体现输出 bytes/token 的降低。
